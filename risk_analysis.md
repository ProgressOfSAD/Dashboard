
# 图书馆系统风险分析
注：影响值 = 可能性 *（成本损失 + 进度损失 + 性能损失）

## 1.	需求风险
### 1.1 对系统需求识别不够充分，有遗漏
 - 集中在管理员系统对用户请求的相应处理操作上。
 
----------

 - 可能性：0.5 
 - 成本损失：0.3 
 - 进度损失：0.4 
 - 性能损失：0.5 


----------


 - 影响值：0.6 
 - 等级：高 
 - 策略：积极应对


----------

 - 预防：加强交流，对两端系统的需求进行对接检查

 
### 1.2 系统需求变动 
- 非工业项目，需求简单，固定，变动可能性不大

----------


 - 可能性：0.1  
 - 成本损失：0.5  
 - 进度损失：0.6
 - 性能损失：0.4


----------


 - 影响值：0.15  
 - 等级：低  
 - 策略：视情况决定是否应对


----------
- 预防： 在早期将需求分析详尽

 
## 2. 计划和控制风险
### 2.1 对项目进度估算不够充分
- 主要可能的情况是对各个阶段实现估算时间过少，导致后续阶段无法按时进行，产品无法按时交货


----------


 - 可能性：0.3 
 - 成本损失：0.2 
 - 进度损失：0.6 
 - 性能损失：0.2


----------


 - 影响值：0.3 
 - 等级：中 
 - 策略：应对


----------
预防：在估算时间时更多以实际情况预测，而不是主观臆断

 
### 2.2 缺少严格的变更控制和版本控制
- 代码混乱，无法回溯，bug增多。不严格使用git，对git使用程度不够

----------


 - 可能性：0.4
 - 成本损失：0.3
 - 进度损失：0.7
 - 性能损失：0.2


----------


 - 影响值：0.48
 - 等级：中
 - 策略：应对


----------

 - 预防：严格，频繁的使用git，记录下每个版本

 
### 2.3 缺少大量的历史数据作为参考

 - 以往没有人写图书馆系统，参考资料过少。


----------


 - 可能性：0.7
 - 成本损失：0.2
 - 进度损失：0.7
 - 性能损失：0.5



----------

 - 影响值：0.98
 - 等级：很高
 - 策略：重点控制


----------
预防：平行参考其他成熟的系统，取其简化的一部分。

 
### 2.4 对项目资源估算不够充分
 - 服务器等


----------


 - 可能性：0.4
 - 成本损失：0.7
 - 进度损失：0.7
 - 性能损失：0.6



----------


 - 影响值：0.8
 - 等级：高
 - 策略：积极应对


----------

 - 预防：提前进行资源估计，各个阶段进行资源重新评估

 
### 2.5 没有完善全面的项目计划
- 项目开展随心所欲，导致前一阶段任务没有完成无法开展后续


----------


 - 可能性：0.6
 - 成本损失：0.3
 - 进度损失：0.8
 - 性能损失：0.6
 


----------


 - 影响值：1.02
 - 等级：很高
 - 策略：重点控制


----------
 - 预防：严格按照tower给出的项目计划进行项目开展。

 
## 3.	技术风险
### 3.1 项目使用的是以前未曾使用的新技术
- 前端框架vue，后端框架Django等


----------


 - 可能性：0.4
 - 成本损失：0.1
 - 进度损失：0.3
 - 性能损失：0.1


----------


 - 影响值：0.2
 - 等级：低
 - 策略：视情况决定是否应对


----------
 - 预防：给技术学习预留足够的时间。

 
## 4.	用户风险
### 4.1 用户不重视项目管理
- 项目虽有计划但并未严格执行。

----------


 - 可能性：0.3
 - 成本损失：0.3
 - 进度损失：0.6
 - 性能损失：0.4


----------


 - 影响值：0.39
 - 等级：高
 - 策略：积极应对


----------
- 预防：制定严格的项目管理，并选择合适的管理人选

### 4.1 用户对该项目的目标和需求不清晰
 - 用户不知道项目的具体作用

----------


 - 可能性：0.2
 - 成本损失：0.5
 - 进度损失：0.8
 - 性能损失：0.7


----------


 - 影响值：0.4
 - 等级：中
 - 策略：应对


----------
 - 预防：积极交流

## 5.	团队风险
### 5.1 团队内部人员难以沟通
 - 沟通渠道小，沟通没有即时反馈

----------


 - 可能性：0.2 
 - 成本损失：0.2 
 - 进度损失：0.9 
 - 性能损失：0.8 


----------


 - 影响值：0.38 
 - 等级：中 
 - 策略：应对


----------
 - 预防：建立交流群，对于新消息开启显眼的提醒。

### 5.2 团队工作效率低下
 - 士气低落，困难攻克效率低，工作效率低

----------


 - 可能性：0.5 
 - 成本损失：0.6 
 - 进度损失：0.9 
 - 性能损失：0.5 


----------


 - 影响值：1.0 
 - 等级：很高 
 - 策略：重点控制


----------
- 预防：进行适当的娱乐或收敛
